## 警告项（建议但不阻塞）

- [ ] **维度：稳定块 / 移动块（矛盾）**
  - 问题：稳定块 6 写"通过 Claude Agent SDK 调用"，而移动块 1 明确"Claude Agent provider/SDK 选择"未定（Anthropic / OpenAI / provider 抽象三选一），§9 也说将在 tech-design 决策。若按字面结晶，REQ 会把一个未定的决策焊死。
  - 建议：把稳定块 6 改为"通过统一的 agent adapter 接口调用（具体 provider/SDK 在 tech-design 决策）"，与 §6.2.2 的接口契约保持一致。

- [ ] **维度：稳定块（retries/onError 适用范围）**
  - 问题：§2.5 说"每个节点可配置重试次数"，稳定块 7 说"节点级配置"，但 §6.2.3 只有 Claude Agent 的 config schema 展示了 `retries`/`onError`。Trigger 几乎不会失败，Condition 的表达式异常是否走重试？结晶时无法确定 REQ 该覆盖哪些节点类型。
  - 建议：明确错误处理配置适用于哪些节点（建议：所有可执行节点统一 schema，或显式声明仅 Agent/Condition）。

- [ ] **维度：稳定块 8（"agent 调用详情/完整记录"不可验收）**
  - 问题：稳定块 8 要求日志包含"agent 调用详情"，用户故事 8 说"完整记录"——具体指什么？完整 prompt/响应文本？模型、耗时、token 用量？写不出断言。
  - 建议：列出一个最小字段清单（如：最终 prompt、输出文本、模型、错误信息），其余显式归入范围外或移动块。另外注意：完整 prompt 可能含敏感信息，建议明确日志是否需脱敏。

- [ ] **维度：稳定块 3（"简单校验"不可验收）**
  - 问题：变量类型"用于 UI 提示和简单校验"——校验发生在何时（保存时/运行时）？类型不匹配的行为是什么（警告/阻止保存/fallback）？"不做严格运行时类型检查"划了下界，但上界模糊。
  - 建议：补一句具体行为，例如"保存配置时校验默认值与声明类型一致，不匹配则阻止保存并提示"。

- [ ] **维度：用户故事（痛点覆盖率未显性）**
  - 问题：痛点中"表达式写错只能在运行时才发现"，但解决方案只承诺表达式输入框 + 变量选择器，设计期校验挂在移动块 4。若移动块 4 最终决策为"不做校验"，该痛点在本 story 实际未被解决，但 PRD 没有显性说出这一点，验收时容易产生预期差。
  - 建议：在移动块 4 或 §9 中显式声明：本 story 是否承诺解决"表达式错误提前发现"，若不承诺则列入范围外。

- [ ] **维度：用户故事（缺少引用失效边界）**
  - 问题：覆盖主路径和失败路径，但缺一个关键边界：上游节点被删除/重命名输出变量后，下游 `{{n1.result}}` 或条件表达式中的引用变成悬空引用，系统的行为是什么（保存时告警？运行时报错？替换为空字符串）？这在"用户无需记忆节点 ID"的承诺下尤其容易发生。
  - 建议：补一个用户故事或在稳定块中声明悬空引用的处理策略（至少声明在运行时的行为）。
